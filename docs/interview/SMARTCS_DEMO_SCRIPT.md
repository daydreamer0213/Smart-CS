# SmartCS 3 分钟演示稿

## 演示定位

SmartCS 是面向企业内部员工的多租户 HR 服务 Agent 后端样板。我要展示的不是模型“会不会聊天”，而是它能否在租户、身份、文档权限和人工处理边界内完成一个可复现闭环。

启动方式见：[本地 HR Agent 演示手册](../operations/local-hr-agent-demo.md)。演示脚本使用虚构的北辰科技数据；运行时生成临时凭据，不显示任何密钥或密码。

评测命令、报告字段和排障见：[M2-5 RAG 评测运行手册](../operations/rag-evaluation-m2-5.md)。这条评测路径不调用 FastGPT、LLM 或 LLM judge。

## 三分钟讲解顺序

1. **先展示可重复的检索证据，约 30 秒**
   - 运行 M2-5 评测命令，打开报告中的 `summary`、`retriever_contributions` 和 `failed_query_ids`。
   - 当前记录为：8 个已索引 fixture、11 个 curated facts-only chunks、12 条 golden queries、`top_k=3`；Recall@3 `11/12 = 91.67%`、MRR `91.67%`、已召回来源 provenance `100%`，gate 为 `passed`。
   - 贡献统计是 BM25 11、vector 0，失败项是 `payroll-contact`。我会主动说明它不是 100%：该中文问题与受信邮箱事实没有词法重合，而 HashEmbedding 是非语义的离线向量通路。保留失败样本比修改问题或制造关键词更诚实。
   - 这证明项目有检索回归门禁，但不把结果包装成混合语义检索质量、通用 PDF 准确率或生产 SLA。

2. **服务可用，约 15 秒**
   - 运行脚本先请求 `/health`，返回 `200`。
   - 这说明当前实例、数据库与基础依赖可以进入演示，而不是只展示静态截图。

3. **身份和租户，约 20 秒**
   - 脚本创建北辰科技 HR 租户，以及 owner、HR admin、employee 三个 JWT 身份。
   - 后端在每个路由上带着租户和角色校验，员工身份不是前端标签。

4. **受众文档、审批发布与制度回答，约 45 秒**
   - owner 上传一份仅对 employee 受众开放的虚构年假制度文档，接口返回 `201`、`ready` 和 `pending_review`；此时员工仍检索不到。
   - owner 审批后 family 的 current 指针才切换，接口返回 `approved + current`。
   - employee 询问“北辰科技年假如何计算？”，HR Agent 返回 `200`；回答文本中出现 `[source:<id>]`，同时 `sources` 返回对应的结构化来源（含 `source_id`）。
   - 随后调用 reindex，展示相同业务版本产生下一索引代次；新代次完整成功后才替换 current。
   - 这证明回答来自员工有权读取且经过审批的当前文档，而不是把制度写死在提示词里。

5. **例外转人工，约 35 秒**
   - employee 提出“海外派驻期间的年假例外，请转 HR 人工处理”。
   - 当制度没有覆盖例外时，Agent 不编造规则，而是返回 `pending_handoff` 草稿。
   - 此时没有正式工单，员工仍掌握是否提交的决定权。

6. **确认后建单与 HR 生命周期，约 45 秒**
   - employee 调用确认接口后，草稿变成 `open` 的正式 HR 支持请求；接口带幂等键，重复确认不会产生重复工单。
   - HR admin 在管理队列中找到它，先设为 `assigned`，再设为 `resolved`。
   - 这个状态变化由后端路由和角色校验控制，不由模型直接改库。

7. **员工可见状态与隔离，约 30 秒**
   - employee 调用 `/hr-support/me`，看到自己这张请求已经 `resolved`。
   - 脚本再创建另一个虚构租户，拿北辰 employee 的 JWT 访问对方租户，得到 `403`。
   - 因此，租户隔离是 API 合同，不是页面筛选。

## 收束讲法

> SmartCS 的价值是把“制度问答”放进受控服务链路：先认证、再按受众检索、回答附来源；遇到例外只生成待确认草稿，确认后才形成 HR 可处理的请求，并且全程有租户和角色边界。这是企业内部 AI 助手比普通聊天机器人更需要的工程能力。

**简历 bullet：** 构建 HR 知识库 RAG 的分层质量门禁：解析层覆盖 OCR/结构化分块/来源元数据，检索层以 12 条 golden queries 验证 Recall@3、MRR、来源溯源及多租户受众边界，并保留失败 query 定位证据。

**面试讲法：** M2-2 parser gate 和 M2-5 retrieval gate 验证不同问题。前者证明固定语料的解析、分块和页级来源；后者验证真实治理检索边界。当前 `payroll-contact` 没有被命中，所以 Recall@3 是 91.67% 而不是 100%；我不会用 HashEmbedding 的通路验证冒充语义检索效果，接入真实 embedding 后仍沿用同一门禁比较。

**可演示路径：** 评测报告的失败项 -> 已审批制度的带页码引用 -> 制度未覆盖例外时的待确认转人工 -> 员工确认后 HR 工单状态与跨租户 `403`。

## 现场异常说明

- `503`：模型、提供方、网络或额度不可用。脚本会失败退出，不能把它描述为演示成功。
- 没有来源引用：停止演示，检查文档导入状态、employee 受众和模型输出；不要用人工补写的引用冒充结果。
- 任何日志、截图或问题单都不应包含 API key、JWT、密码或 `.env` 内容。
