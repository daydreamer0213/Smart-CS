# SmartCS 最终求职表达稿

## 项目名称

SmartCS：多租户企业员工 Agent 后端

## 一句话版本

SmartCS 是一个面向企业内部员工场景的 Python AI 后端项目：员工登录后，系统根据角色开放企业知识检索和 CRM Skills；AI 可以准备业务操作，但写入必须经过用户确认、权限复核、幂等处理和审计记录。

## 简历项目描述

### 标准版

独立构建多租户企业员工 Agent 后端 SmartCS，基于 FastAPI、SQLAlchemy、ChromaDB、BM25、LangGraph 和 JWT，实现企业知识 RAG、文档导入、角色化 Skills、CRM 查询、业务操作草稿、确认后写入、审计日志、租户隔离和自动化测试。

### AI Agent 岗位版

构建企业员工 Agent 后端，基于 LangGraph 设计知识检索、CRM 查询和业务操作草稿工具；按 owner/admin/agent/employee 角色动态暴露 Skills，并通过“草稿 -> 确认 -> 写入 -> 审计”流程约束 Agent 的业务操作能力。

### Python 后端岗位版

构建 FastAPI 多租户 AI 应用后端，设计租户、用户、角色、知识库、文档、会话、CRM 客户/线索/任务、操作草稿和审计日志模型，实现 JWT / API Key 双认证、租户隔离、角色权限、幂等确认和回归测试。

### RAG / LLM 应用岗位版

实现企业知识 RAG 后端链路，支持 txt/md/pdf/docx/xlsx 文档导入、解析、分块、ChromaDB 向量检索、BM25 检索、角色化知识可见范围和 Agent 检索工具，并提供 hash embedding 离线模式保证演示稳定。

## 最推荐的 3 条简历 bullet

- 构建 FastAPI 多租户企业员工 Agent 后端，覆盖租户、用户、角色、知识库、文档、CRM、会话、审计日志、JWT 和 API Key 管理。
- 实现企业 RAG + 角色化 Skills，支持文档导入、向量检索、BM25 检索、知识可见范围控制、CRM 查询和统一 `/assistant` 入口。
- 设计受控业务写操作流程，AI 只生成操作草稿，用户确认后后端重新校验并写入，支持幂等确认、重复线索保护和审计记录。

## 面试开场 30 秒

SmartCS 是我用来证明 Python AI 后端能力的项目。它不是单纯聊天机器人，而是一个企业员工 Agent：员工登录后，系统根据角色开放不同 Skills。普通员工可以查企业知识；销售或管理员可以查 CRM，并让 AI 准备线索或任务操作草稿。真正写入数据库必须用户确认，后端重新校验权限并记录审计。这个项目展示的是 Agent 落地到企业系统时需要的权限、数据、工具调用和测试闭环。

## 面试深讲 2 分钟

我把 SmartCS 定位成企业员工 Agent，而不是客服 FAQ。真实企业接入 AI 时，难点不是让模型说话，而是让它在员工身份、租户边界、角色权限和业务数据约束下安全工作。

后端用 FastAPI 和 SQLAlchemy。认证上实现 JWT 和 API Key 两条路径，owner 可以创建租户，owner/admin 可以创建同租户用户，employee 只能使用知识 Skill，agent/admin/owner 可以使用 CRM Skill。所有后台和 Skill 调用都要校验租户边界。

AI 能力分两层：第一层是企业知识 RAG，支持文档导入、分块、ChromaDB、BM25 和知识可见范围；第二层是企业 Agent 工具调用，能查知识、查本地 CRM 客户概览，并准备创建线索、更新线索、创建跟进任务的草稿。为了避免 LLM 直接改业务数据，我把写操作设计成确认式流程：Agent 只能生成草稿，用户确认后，后端重新校验权限、处理幂等键、写入数据库并记录审计日志。

这个项目对我求职的价值是，它证明我不只是会调用模型，还能处理企业 AI 应用真正落地时会遇到的身份、权限、RAG、工具调用、业务写入和测试问题。

## 高频追问回答

### 这个项目对企业有什么意义？

企业真正需要的是可治理的 AI 工作流。SmartCS 解决的是：员工身份要可信，知识和 CRM 数据要按角色开放，Agent 不能绕过权限直接写库，所有业务写入要确认、可重放、可审计。这比“能聊天”更接近企业内部 AI 助手的真实要求。

### 你最核心的工作是什么？

最核心的是把项目从问答后端推进成企业员工 Agent 后端：补了 JWT 身份体系、多租户边界、角色化 Skills、统一 `/assistant` 入口、本地 CRM demo、确认后写入、审计日志、离线演示模式和关键测试。

### 最值得讲的技术点是什么？

我会讲“Agent 工具能力如何被后端权限约束”。比如 employee 只能获得 `knowledge.search`，agent/admin/owner 才能获得 `crm.read` 和 `crm.prepare_change`。写操作不会由 LLM 直接执行，而是先生成草稿，再由确认接口重新校验和写入。

### 遇到过什么安全问题？

JWT 初版有一个边界风险：如果只靠请求体指定 `tenant_slug` 和 `role`，攻击者可能尝试给已有租户注册 admin。后来我用测试复现这个问题，把根因修到注册逻辑里：已有租户创建 admin/agent/employee 必须由同租户 owner/admin JWT 或同租户 API Key 授权。

### 为什么加 hash embedding？

真实 embedding 依赖外部额度和网络，面试演示容易不稳定。hash embedding 不是生产方案，而是为了稳定展示工程链路：租户、权限、知识创建、文档导入和统一 Agent API 都能在本地跑通。生产环境仍然使用真实 embedding provider。

### 这个项目还不是生产 SaaS，怎么说明边界？

我会明确说它是求职样板项目，不是已经商业上线的 SaaS。它证明的是 Python AI 后端工程能力。真正生产化还需要邀请流、CI/CD、部署加固、密钥管理、监控告警、更完整观测链路和真实业务系统集成。

## 不建议这样说

- 不要说“这是完整商业 SaaS”，应说“企业 AI 后端工程样板”。
- 不要说“实现了生产级 CRM Agent”，应说“实现了本地 CRM 场景下的受控操作 MVP”。
- 不要承诺“回答准确率很高”，应说“项目重点是可治理 RAG 和工程闭环，答案质量取决于业务文档和模型配置”。
- 不要把 UI 当主卖点，主卖点是 Python 后端、RAG、Agent、权限边界、确认后写入和测试。

## 最后的收束句

SmartCS 对我的求职价值是：它把 RAG、Agent 和 Python 后端工程放进一个能跑、能测、能演示的企业员工场景里，证明我不仅会调用模型，也能处理企业 AI 应用落地时绕不开的身份、权限、数据、业务写入和审计问题。
