# SmartCS 求职作品集证据素材包设计

**日期：** 2026-07-20

**状态：** 待用户审阅

## 2026-07-20 范围修正

用户确认 SmartCS 任务应只向作品集任务提供项目素材，不重复实现作品集前端。因此，本文后续关于 SC-01 至 SC-04 静态 HTML 证据板的制作规格已被实施计划中的 E2 中立素材契约取代。

当前 E2 交付 `project-facts.json`、`claims.json`、`portfolio-copy.md` 和消费说明；不交付 HTML、CSS、React 组件或最终视觉。真实图像素材在 E3 从实际演示视频和关键帧中产生，页面排版、样式和交互仍由作品集任务独立决定。

## 决策摘要

SmartCS 将作为 AI 应用作品集中的第一个公开项目。素材不采用大面积遮挡截图、虚构产品后台或通用“企业 AI”效果图，而是使用虚构 HR 演示数据运行真实 SmartCS 代码，并让每一项可见结论都有可追溯来源。

本阶段不修改仍在调整的作品集前端。SmartCS 先独立产出证据素材包；等素材经用户批准、作品集前端具备集成条件后，作品集会话只能消费已批准的导出文件、文案和项目边界。

## 目标

让 HR 或面试官在大约一分钟内通过直观材料确认 SmartCS 是能运行的 Python AI 后端，而不是概念页面：

1. HR 制度文档经过质量门禁和审批后才对员工生效。
2. Agent 回答包含员工有权查看的来源引用。
3. 制度未覆盖的例外进入“草稿、员工确认、HR 处理”闭环。
4. 租户和角色边界由后端 API 强制执行。
5. 检索质量和回归测试有可复现结果，同时保留失败样本和限制。

素材包服务求职展示，不新增 SmartCS 产品能力，也不宣称项目已经生产部署。

## 项目分工

SmartCS 在作品集中主要证明独立完成 Python 企业 AI 后端的能力，与其他项目形成互补：

| 项目 | 主要证明 |
| --- | --- |
| SmartCS | 企业 AI 后端、可治理 RAG、多租户权限和自动化测试。 |
| KnowledgeFlow | LangGraph 多 Agent 编排和质量闭环。 |
| 岗位投递助手 | 从个人真实流程出发持续使用的 AI 自动化工具。 |
| DocMind | 企业项目协作经历，不展示私有代码或伪造系统截图。 |

## 证据原则

### 真实运行，虚构业务数据

- 只使用现有演示创建的虚构“北辰科技”租户、用户、制度和 HR 请求。
- 所有业务画面显著标注 `公开演示数据`。
- API 状态、评测结果和流程顺序必须来自实际运行，不画出未观察到的结果。
- 模型、网络、额度或提供方故障属于采集失败。不能把 `503` 做成成功截图，也不能手工补写缺失引用。

### 展示层和原始证据成对存在

每项整理后的文案或画面都必须关联一份不含秘密的原始记录。展示层可以改善排版、分组和说明，但不能修改状态、指标、检索器贡献、失败问题或流程顺序。

### 限制必须直接展示

- `Recall@3 = 91.67%`、`MRR = 91.67%` 是固定 curated corpus 的离线检索回归结果，不是生产 SLA。
- `provenance accuracy = 100%` 只针对已经召回的来源块，不代表 12 个问题全部命中。
- 当前记录的贡献是 BM25 `11`、vector `0`；HashEmbedding 是非语义实现，不能包装成语义检索质量。
- SmartCS 是企业 AI 应用工程样板，不是已经上线运营的商业 HR SaaS。

## 事实来源

| 证据 | 唯一事实来源 |
| --- | --- |
| 项目定位与架构 | 采集 Git commit 对应的 `README.md`。 |
| HR 实时闭环 | E3 隔离运行中的 `preflight.stdout.utf8.txt`，由 `scripts/demo_enterprise_flow.py` 成功运行生成。 |
| 演示验收合同 | `docs/operations/local-hr-agent-demo.md`。 |
| 检索指标 | E3 隔离运行中的新鲜 `rag-evaluation-narrative.json`。 |
| 指标解释与限制 | `docs/operations/rag-evaluation-m2-5.md`。 |
| 回归测试结果 | 同一 Git commit 下重新执行 `python -m pytest -q` 的输出。 |

素材清单中的每一项都要记录 Git commit、采集时间、命令或接口、原始来源路径和从来源提取的结论。

## 存储结构

生成文件和大文件全部放在 D 盘：

```text
D:\DevData\smartcs\portfolio-evidence\
  materials\ # E2 结构化文案和事实契约
  e3\        # 隔离运行、脱敏日志、评测结果和视频清单
  masters\   # 视频母版
  exports\   # Web MP4、封面、字幕和关键帧
  drafts\    # 不交接的旧方案和失败采集
```

视频母版、原始报告、数据库、Chroma 数据和采集缓存都不复制到 C 盘上的作品集目录。用户批准后，作品集会话只能复制选中的网页轻量导出文件，或使用真实的公开资源 URL；公开网页不能引用本机 `D:` 路径。

## 已取代的静态证据板草案（保留备查，不实施）

以下 SC-01 至 SC-04 仅保留最初设计思路，不属于当前交付。E2 只交付结构化材料，E3 的 SC-05 已吸收其中必要的证据表达；不得据此重新开发静态页面或第二套前端。

### SC-01：项目总览证据板

**目的：** 不阅读 README 也能理解项目类型和完整闭环。

**规格：** 1600 x 900，浅色中性背景，提供 WebP 和 PNG 母版。

**画面内容：**

- 标题：`SmartCS 企业人事知识 Agent`。
- 一句话：在一个 FastAPI 后端中完成制度治理、来源引用、受控转人工和租户边界。
- 五步架构：`JWT 身份 -> 文档治理 -> 混合检索 -> 带来源回答 -> 确认后转 HR`。
- 四个证据标签：多租户后端、文档治理、检索评测、自动化回归。
- 显著标记：`公开个人项目`、`公开演示数据`。

这是一张由源码和接口支撑的架构图，不冒充运行截图；每个模块必须能映射到现有代码或 API。

### SC-02：制度治理与带来源回答证据板

**目的：** 同时证明上传知识不会立刻暴露给员工，以及 Agent 回答确实连接到了审批后的有权来源。

**规格：** 1600 x 1000，WebP 和 PNG 母版。

**画面内容：** 上半部分使用同一次成功演示的真实状态，按顺序展示：

1. 上传后返回 `ready + pending_review`。
2. 该快照尚未成为员工可检索的 current 版本。
3. owner 审核后返回 `approved + current`。
4. reindex 生成下一 `index_generation`，完整成功后才切换 current。

下半部分展示：

- 员工问题：`北辰科技年假如何计算？`。
- 一段实际成功回答，必须包含 `[source:<id>]`。
- 对应结构化来源条目，展示 source ID 和当次响应实际具备的页码或章节信息。
- 说明：只有审批通过且受众角色匹配后，该来源才能进入回答。

画面显示实际状态字段和简短演示 ID。随机 UUID 可以折叠，但必须注明 `随机演示 ID 已折叠`，不能改变状态和顺序。如果实际模型响应没有引用，本次运行不能产出 SC-02，严禁手工添加引用。

### SC-03：受控转人工与隔离证据板

**目的：** 通过受控工具调用和后端状态变化体现 Agent 属性，而不是画一条开放式工作流。

**规格：** 1600 x 1000，WebP 和 PNG 母版。

**画面内容：**

```text
海外派驻年假例外
  -> pending_handoff 草稿
  -> 员工确认
  -> open
  -> HR assigned
  -> HR resolved
  -> 员工看到 resolved
```

右下角单独展示跨租户请求返回 `403`。画面明确说明：模型只产生草稿，确认和工单状态变化由后端 API 控制。

### SC-04：工程验收证据板

**目的：** 用一张容易读懂的图同时展示诚实的 RAG 评测结果和 Python 后端回归证据。

**规格：** 1600 x 900，WebP 和 PNG 母版。

**画面内容：** 左侧为检索评测：

- 8 个已索引 fixture，11 个 curated facts-only source chunks。
- 12 条 golden queries，`top_k=3`。
- Recall@3 `11/12 = 91.67%`，MRR `91.67%`。
- 已召回来源 provenance `100%`。
- BM25 贡献 `11`，vector 贡献 `0`。
- 保留的失败问题：`payroll-contact`。

右侧为自动化回归：

- 新鲜全量测试结论：`398 passed, 4 skipped`。
- 测试领域：认证、租户隔离、文档解析和治理、检索、Agent 工具、转人工生命周期、数据库迁移。
- 采集 Git commit 和时间。

失败问题和 HashEmbedding 的非语义限制必须放在主画面中，不能藏在细小脚注里。现有 Jieba `pkg_resources` 弃用警告不能被写成测试失败，也不能宣称输出完全无警告。这张图是辅助证据，不作为项目卡第一张图。

### SC-05：60 至 90 秒真实演示视频

**目的：** 把真实 API 运行和检索评测中的分散证据，组织成一条可快速审阅的企业业务链路。

**规格：** 1920 x 1080 MP4 母版放在 `masters`，网页版本放在 `exports`。

**分镜：**

| 时间 | 画面 |
| --- | --- |
| 0-7 秒 | 项目定位和五段后端处理链。 |
| 7-16 秒 | 知识生效、Agent 权限、租户边界三类企业问题。 |
| 16-26 秒 | 文档上传、审批发布和重新索引代次。 |
| 26-39 秒 | 员工提问、真实回答节选和一致的来源 ID。 |
| 39-50 秒 | 转人工草稿、员工确认、HR 接单和解决。 |
| 50-58 秒 | 跨租户请求真实返回 `403 tenant_access_denied`。 |
| 58-69 秒 | 检索指标、检索器贡献和失败问题。 |
| 69-79 秒 | 全量测试范围、真实限制和非生产 SLA 声明。 |

视频必须来自一次真实成功运行，以明确标注的“真实运行脱敏回放”呈现，不是产品界面或终端录屏。画面可以重新排版已核验的状态、回答、引用和指标，但不能改写结果、模拟产品交互或隐藏失败项。配音可选，必须提供简洁中文字幕。首次作品集发布如果没有可靠的公共视频托管，可以只使用经批准的封面图，不引用本机视频路径。

## 采集与排版方法

1. 在 D 盘创建全新演示数据库，并使用受支持的在线 Alembic 迁移。
2. 本地启动 SmartCS，使用虚构北辰科技数据运行 `scripts/demo_enterprise_flow.py`。
3. 由 Python 直接保存 UTF-8 stdout 和结构化评测结果；禁止采集 `.env`、Authorization header、JWT、密码、API key、`storage_key` 和物理文件路径。
4. 在同一 Git commit 下重新执行检索评测和全量 pytest。
5. 使用 Pillow 将实际采集值排成八个中性叙事画面，并用已有 FFmpeg 编码；不引入前端框架，也不使用图片生成模型伪造产品界面。
6. 从最终视频抽取八个关键帧，与原始来源逐项核对，并把来源哈希、输出哈希和媒体参数写入 `video-manifest.json`。

这些叙事画面只是证据排版，不是第二套作品集前端或 SmartCS 产品界面。它们使用中性版式和轻微转场，不增加项目运行时依赖。

## 隐私与处理规则

允许展示：

- 虚构北辰科技名称和制度。
- `example.test` 邮箱。
- 实际的合成业务状态和公开评测数据。
- 明确标记后折叠随机演示 UUID。

禁止展示：

- API key、JWT、密码、Cookie、Authorization header 或 `.env` 内容。
- 真实员工、企业、客户、租户或面试信息。
- 本机用户目录和与项目无关的桌面内容。
- DocMind 或任何雇主的真实企业文档。
- 虚构客户、上线量、延迟、使用量、准确率或 SLA。
- 把 AI 生成的后台界面当成产品运行结果。

## 批准后的实施阶段

### E1：复现并采集原始证据

- 只检查模型配置是否齐全，不读取或打印秘密值。
- 运行新鲜实时演示、检索评测和全量测试。
- 原始证据保存在 `D:\DevData\smartcs\portfolio-evidence\e3\<run-id>` 隔离运行目录。
- 实时回答缺少来源时停止，不继续制作成功素材。

### E2：制作和核验结构化交接材料

- 产出 `project-facts.json`、`claims.json`、`portfolio-copy.md` 和消费说明。
- 对照原始证据核验数字、限制和禁止宣传项。
- 不开发 HTML、CSS、React 组件或最终作品集视觉。

### E3：生成和核验真实运行证据短片

- 从真实成功运行和新鲜检索评测生成 SC-05 脱敏证据回放。
- 生成视频封面和网页轻量版本，母版不复制到 C 盘。
- 抽取八个成片关键帧，确认没有秘密、本机路径、内容遮挡或空白画面。

### E4：用户审阅与作品集交接

- 本地展示全部证据板和视频，让用户选择。
- 只针对用户指出的具体可读性或可信度问题调整证据表达。
- 等作品集前端具备集成条件后，再发送批准后的清单和文件。

## 不做清单

- 不接 HRIS，不增加 Agent 工具，不开发 SmartCS 前端或产品后台。
- 不修改独立的作品集前端。
- 不采集真实 HR 或雇主数据。
- 不试图在一张图里展示全部 SmartCS 功能。
- 不制作遮挡严重、无法核验的“企业截图”。
- 现有本地能力能够完成时，不新增视觉框架、设计依赖或视频工具。

## 验收标准

- 每张必做证据板只表达一个主要结论，不依赖阅读代码墙。
- 所有数字和状态都能追溯到原始记录和 Git commit。
- 业务画面显著标记为公开虚构演示数据。
- 不出现秘密、企业私有材料、本机用户路径或无关桌面内容。
- 只有真实模型返回带来源回答后才存在 SC-02。
- 视频状态和边界与经批准证据板一致。
- 在不修改作品集前端的情况下，可以单独审阅素材包。
- 交接清单包含最终说明、alt text、宽高比、指标、限制和禁止宣传内容。
